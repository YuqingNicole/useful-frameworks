## 演绎法（强制）

修改 src/ 时，按顺序执行：
1. 查 .deductive/acs/rules.json → 向用户展示相关规则
2. 用业务语言写验收要点（3-5条）→ 等用户说"确认"
3. 写 tests/test_ac_*.py（头部 # @covers: 规则ID）+ 写生产代码
4. 跑一次给用户看结果 → 等用户说"对"
5. 用户说"对" → commit；说"不对" → 改了再跑给用户看

不可跳步。未确认不写代码。未验证不 commit。
无匹配规则时告知用户后正常实现。
用户说"跳过演绎"时可直接实现。

### 验收要点规则
用用户能判断对错的业务语言，禁止写技术细节。

### 演示规则
能演示的改动（文案生成、评分、prompt内容）：用真实数据跑一次，展示实际输出。
无法演示的改动（数据库、算法内部、基础设施）：诚实告知用户"这个没法跑给你看，由 AC 测试自动守卫"，不要伪造演示。

### 意图识别规则
系统已配置 UserPromptSubmit hook 做意图识别。
当 hook 返回 UNCERTAIN 时，不要视为确认。必须追问用户，说清楚用户在犹豫什么，帮用户理解后再请求明确确认。
"应该对吧""算是吧""嗯...""可能" = 不确定 = 必须追问。

### 示例
  用户："实现 R1 温度差异化"
  AI：
    步骤1："R1 的意思是：不同情绪方向的内容，生成风格应该不同。"
    步骤2："验收要点：
           1. 温暖类内容 → 文案稳定温柔
           2. FOMO类内容 → 文案大胆有冲击
           3. 同一产品不同情绪 → 风格明显不同
           这是你要的效果吗？"
    用户："确认" → 继续
    用户："应该对吧" → AI追问："你对哪一点不确定？我再解释一下。"

    步骤3：写测试 + 写代码
    步骤4（能演示）："我用'18K金项链'跑了两次：
           温暖方向 → '这条链子陪了我整个冬天...'
           FOMO方向 → '姐妹们冲！断货三次终于补到...'
           效果对吗？"
    步骤4（无法演示）："这个改动是数据库层的，没法跑给你看效果。
           它由 test_ac_006 自动守卫，如果未来被改坏，commit 时会被拦住。"
